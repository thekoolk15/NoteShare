Unit 1: Introduction to Databases

15 contact hours · 3 major topics

Overview of Databases
Fundamentals▼
Textbook Definition

A database is an organized, persistent collection of logically related data that models some aspect of the real world. A Database Management System (DBMS) is a software system that enables users to define, create, maintain, and control access to a database.

📚 Think of a database like a super-organized digital filing cabinet. Instead of scattered papers, all data is neatly stored. The DBMS is the filing cabinet manager — it lets you add, search, update, and delete files. Example: Facebook stores all your posts, photos, and friends in a database. The DBMS is what lets you instantly search for a friend among billions of users.

Three-Level ANSI-SPARC Architecture

Textbook Definition

The ANSI-SPARC architecture defines three levels of abstraction: External Level (user views), Conceptual Level (logical structure of entire database), and Internal Level (physical storage structure). This separation provides data independence.

External Level (View Level)

What individual users see. Each user/app gets a customized view. E.g., HR sees employee salaries; Sales sees customer orders only.

Conceptual Level (Logical Level)

The community view. Describes what data is stored and the relationships. DBA works at this level. Hides physical storage details.

Internal Level (Physical Level)

How data is physically stored on disk — file organization, indexes, access paths, block sizes, compression. Hidden from users entirely.

🏢 Imagine a bank. A customer sees their own account (External). The bank manager sees all accounts and relationships (Conceptual). The computer engineers handle hard drives and file formats (Internal). The beauty: when engineers upgrade the hard drives (internal change), customers don't notice anything. That's data independence!

Data Independence

  • Physical Data Independence: Ability to change the internal/physical schema without affecting the conceptual schema. E.g., changing from HDD to SSD without rewriting queries.
  • Logical Data Independence: Ability to change the conceptual schema without affecting external schemas/user applications. E.g., adding a new column to a table without breaking existing apps.

Integrity Constraints

Entity Integrity

No primary key attribute can be NULL. Every row must be uniquely identifiable. A student must have a StudentID — it cannot be empty/null.

Referential Integrity

A foreign key value must either be NULL or match an existing primary key in the referenced table. You can't enroll a student in a course that doesn't exist.

Schema vs Instance

  • Schema: The structure/blueprint of a database — table names, column names, data types, constraints. Rarely changes.
  • Instance: The actual data in the database at a particular moment in time. Changes constantly.

Advantages of DBMS

  • Reduced data redundancy (no duplicate data in multiple files)
  • Data consistency (one update reflects everywhere)
  • Data sharing (multiple users/apps access simultaneously)
  • Data security and access control
  • Backup and recovery mechanisms
  • Data integrity enforcement via constraints

Disadvantages of DBMS

  • High initial cost of software and hardware
  • Complexity — requires trained personnel (DBA)
  • Performance overhead compared to flat-file systems for simple tasks
  • Single point of failure (if DBMS crashes, all apps fail)

DBA (Database Administrator) — Responsibilities

  • Defining the conceptual and internal schemas
  • Managing user accounts and security privileges
  • Monitoring performance and tuning queries
  • Planning and implementing backup/recovery strategies
  • Ensuring data integrity and availability
  • Liaising between users and the DBMS system
Data Models
Core Concepts▼
Textbook Definition

A data model is a collection of concepts used to describe the structure of a database — the data types, relationships, and constraints that apply to data. It provides a formal means to represent real-world entities and their interactions.

Basic Building Blocks

  • Entity: A real-world object or concept — e.g., Student, Course, Employee
  • Attribute: A property or characteristic of an entity — e.g., Student has Name, Age, RollNo
  • Relationship: An association between entities — e.g., Student "enrolls in" Course
  • Constraint: A rule that limits valid data — e.g., Age must be > 0

Degrees of Data Abstraction

  • Physical Level: How data is actually stored (bits, bytes, files, indexes)
  • Logical Level: What data is stored and relationships between data (tables, columns)
  • View Level: Portion of database relevant to a particular user or application

Entity-Relationship (ER) Model — Deep Dive

Core ER concepts

The ER model represents the database design visually using entities (rectangles), attributes (ellipses), relationships (diamonds), and connecting lines with cardinality notations.

  • Strong Entity: Has its own primary key (e.g., Student with StudentID)
  • Weak Entity: Depends on a strong entity for identification (e.g., Dependent of an Employee)
  • Simple Attribute: Cannot be divided (e.g., Age)
  • Composite Attribute: Can be divided into sub-attributes (e.g., Name → FirstName + LastName)
  • Derived Attribute: Can be computed from other attributes (e.g., Age derived from DateOfBirth)
  • Multi-valued Attribute: Can have multiple values (e.g., Phone numbers)

Mapping Constraints (Cardinality)

  • 1:1 — One-to-One: One employee has one parking spot
  • 1:N — One-to-Many: One department has many employees
  • M:1 — Many-to-One: Many employees work in one department
  • M:N — Many-to-Many: Many students enroll in many courses

🎓 Think of an ER diagram as a blueprint of a building before construction. Before building the database, you draw a map: boxes are things (Student, Teacher), ovals are their details (Name, Age), and diamonds show connections (Student TAKES Course). The cardinality is like saying "one teacher can teach many classes, but one class has only one teacher." Once the blueprint is approved, you then convert it to actual tables.

Business Rules

Business rules are statements that define or constrain some aspect of the business. They are the source of constraints in the ER model. Example: "A student cannot enroll in more than 6 courses per semester" → becomes a constraint in the database schema.

Evolution of Data Models

  • 1960s: Hierarchical (IMS by IBM) and Network models (CODASYL)
  • 1970s: Relational model proposed by E.F. Codd (revolutionary — SQL-based)
  • 1980s: Object-Oriented databases (OODBs)
  • 1990s-2000s: Object-Relational databases (ORDBMs)
  • 2000s+: NoSQL (document, key-value, graph, column-store) for big data

UML in Databases

Unified Modeling Language (UML) class diagrams can model database structures. Classes = tables, attributes = columns, associations = relationships. UML is more expressive than ER and supports inheritance (generalization/specialization).

Relational Algebra and Relational Calculus
Query Languages▼
Textbook Definition

Relational Algebra is a procedural query language that uses a set of operations on relations (tables) to produce new relations. It forms the theoretical foundation for SQL. Relational Calculus is a non-procedural (declarative) query language that describes what to retrieve, not how.

🔧 Relational Algebra is like giving a robot step-by-step instructions: "First filter rows, then join two tables, then pick specific columns." Relational Calculus is like describing what you want: "Give me all students who are enrolled in CS courses" — without saying how to find them. SQL is mostly declarative (calculus-style) but optimizers convert it to algebra steps internally.

Cartesian Product×All possible combinations of rows from two tablesCROSS JOIN
Natural Join⋈Join on common attributes, removes duplicate columnsNATURAL JOIN

Examples

σ Age > 20 (Student) -- Select students older than 20 π Name, RollNo (Student) -- Project only Name and RollNo columns Student ⋈ Enrollment -- Natural join on common attribute σ Dept='CS' (π Name,Dept (Employee)) -- Composition of operations

Additional Operators

  • Rename (ρ): Renames a relation or its attributes — ρ(NewName, OldRelation)
  • Theta Join (⋈θ): Join with any condition (not just equality)
  • Equijoin: Join where the condition is equality on specified attributes
  • Outer Join: Left ⟕, Right ⟖, Full ⟗ — preserves non-matching tuples using NULLs
  • Grouping/Aggregation (γ): Groups tuples and applies aggregate functions (COUNT, SUM, AVG)

Relational Calculus

Tuple Relational Calculus (TRC)

Variables range over tuples of a relation. Syntax: {t | P(t)} — "all tuples t such that predicate P(t) is true." Example: {t | t ∈ Student ∧ t.Age > 20} gives all students over 20.

Domain Relational Calculus (DRC)

Variables range over domains (attribute values), not tuples. Syntax: {⟨x,y⟩ | P(x,y)}. Example: {⟨n,a⟩ | ⟨n,a⟩ ∈ Student ∧ a > 20} gives names and ages of students over 20.

Calculus vs Algebra Comparison

AspectRelational AlgebraRelational Calculus
TypeProceduralNon-procedural (declarative)
FocusHow to retrieve dataWhat data to retrieve

Codd's TheoremBoth have equal expressive power — any query in one can be expressed in the other

Computational Capabilities

A query language is relationally complete if it can express all queries expressible in relational algebra. Both TRC and DRC are relationally complete. SQL is also relationally complete (plus it has extra features like aggregation, ordering). Relational algebra/calculus cannot express some queries like transitive closure or recursive queries without extensions.

Functional Dependencies and Normalization
Most Important▼
Textbook Definition

A functional dependency (FD) X → Y holds on a relation R if and only if whenever two tuples of R agree on the attributes X, they must also agree on attributes Y. We say "X functionally determines Y" or "Y is functionally dependent on X."

🔑 If you know someone's StudentID, you automatically know their Name, Age, and Department. So: StudentID → Name, Age, Department. Knowing the StudentID "determines" all other info. This is functional dependency. It's like: knowing your Aadhaar number means you can find your name — Aadhaar# → Name.

Types of Functional Dependencies

Full Functional Dependency (FFD)

Y is fully functionally dependent on X if Y depends on all of X and not on any proper subset of X. Example: {StudentID, CourseID} → Grade. Grade depends on BOTH attributes together.

Partial Dependency

Y depends on a proper subset of a composite key X. Example: {StudentID, CourseID} → StudentName, but StudentName only depends on StudentID alone. Bad! Causes redundancy.

Transitive Dependency (TD)

X → Y and Y → Z, so X → Z (transitively). Example: StudentID → DeptID → DeptName. StudentID transitively determines DeptName through DeptID. Causes update anomalies.

Multi-valued Dependency (MVD)

X →→ Y means that for each value of X, the set of Y values is independent of other attributes. Example: Employee →→ Skill, Employee →→ Project. Leads to 4NF.

Join Dependency (JD)

A relation R satisfies a join dependency ✶[R1, R2, ..., Rn] if R can always be reconstructed by joining its projections onto R1, R2, ..., Rn without loss of information. Relevant for 5NF.

Normalization — Normal Forms

Textbook Definition

Normalization is the process of organizing a relational database to reduce data redundancy and improve data integrity by decomposing relations into smaller, well-structured relations while preserving data and dependencies.

🗂️ Normalization is like decluttering your wardrobe. Bad design: one huge table with everything mixed together (student info + course info + teacher info in one row). When a teacher's phone number changes, you update 100 rows — that's a mess (update anomaly). Normalization splits it into neat separate tables (Students, Courses, Teachers) and links them with foreign keys. Result: change teacher's phone in ONE place, it reflects everywhere.

Normal FormRuleWhat it eliminates
1NF (First)All attributes must be atomic (indivisible). No repeating groups. Every cell holds a single value.Multi-valued attributes, repeating groups
2NF (Second)Must be in 1NF. Every non-key attribute must be fully functionally dependent on the entire primary key (no partial dependencies).Partial dependencies
3NF (Third)Must be in 2NF. No non-key attribute should transitively depend on the primary key. Every non-key attribute depends directly on the primary key.Transitive dependencies
BCNF (Boyce-Codd)Stronger than 3NF. For every FD X → Y, X must be a super key. Handles anomalies 3NF misses with multiple overlapping candidate keys.All FD anomalies not covered by 3NF
4NFMust be in BCNF. No multi-valued dependencies except on super keys.Multi-valued dependencies
5NFMust be in 4NF. No join dependencies unless they follow from candidate keys.Join dependencies

Anomalies (why normalization is needed)

  • Insertion Anomaly: Cannot insert data without also inserting unrelated data
  • Update Anomaly: Updating one fact requires updating many rows; inconsistency if some are missed
  • Deletion Anomaly: Deleting one piece of data accidentally deletes other important data

Decomposition

Splitting a relation into smaller relations. Must preserve:

  • Lossless Join: Original relation can be reconstructed exactly by joining decomposed relations (no spurious tuples)
  • Dependency Preservation: All original functional dependencies can be enforced in the decomposed relations

De-normalization

The intentional introduction of redundancy into a normalized schema to improve read performance. Used in data warehousing and OLAP systems where query speed matters more than update efficiency. Trade-off: faster reads but more storage and harder updates.

Data Security
Security▼
Textbook Definition

Database security refers to the collective measures used to protect and secure a database or database management software from illegitimate use, malicious threats and attacks, and unauthorized access. It encompasses the confidentiality, integrity, and availability (CIA Triad) of data.

Threats to Database Security

  • Unauthorized Access: Users accessing data they shouldn't (privilege escalation)
  • SQL Injection: Malicious SQL code embedded in user inputs to manipulate queries
  • Inference Attack: Deducing sensitive info from non-sensitive data
  • Insider Threats: Authorized employees misusing access privileges
  • Denial of Service (DoS): Overloading the database server to make it unavailable
  • Data Theft/Breach: Unauthorized copying or exfiltration of data
  • Malware: Viruses or ransomware encrypting/destroying database files

Countermeasures

  • Authentication: Verifying the identity of users (passwords, MFA, biometrics)
  • Authorization: Granting appropriate privileges using GRANT/REVOKE in SQL. Views restrict column-level access.
  • Encryption: Encrypting data at rest and in transit (TLS/SSL, AES)
  • Auditing: Logging all database access and modifications for forensic analysis
  • Access Control: DAC (Discretionary), MAC (Mandatory), RBAC (Role-Based)
  • Input Validation: Preventing SQL injection through parameterized queries / prepared statements
  • Backup and Recovery: Regular backups to recover from attacks or failures

🔒 SQL Injection example: A login form takes username/password. If you type username: admin'--, a badly designed system might execute: SELECT * FROM users WHERE username='admin'-- AND password='...' — the -- comments out the password check! Countermeasure: use parameterized queries that treat user input as data, never as SQL code. This is a major real-world threat — countless companies have been breached this way.

Control Structures, Cursors and Views
PL/SQL▼
Control Structures in PL/SQL

PL/SQL (Procedural Language for SQL) extends SQL with procedural constructs including conditional control, iterative control, and sequential control — enabling complex logic within database programs.

Types of Control Structures

Conditional Control

IF-THEN, IF-THEN-ELSE, IF-THEN-ELSIF, CASE statements. Executes code based on conditions.

Iterative Control

LOOP (basic), WHILE-LOOP, FOR-LOOP. Repeats a block of code until a condition is met or a range is exhausted.

Sequential Control

GOTO statement (jumps to a labeled section) and NULL statement (does nothing, used as a placeholder). Generally avoided in structured programming.

-- IF-THEN-ELSE example IF salary > 50000 THEN bonus := salary * 0.2; ELSIF salary > 30000 THEN bonus := salary * 0.1; ELSE bonus := 0; END IF; -- FOR LOOP example FOR i IN 1..10 LOOP DBMS_OUTPUT.PUT_LINE(i); END LOOP;

Cursors

Textbook Definition

A cursor is a named control structure (pointer) in PL/SQL that refers to a private SQL work area where Oracle stores information about the processing of a SELECT or DML statement. It allows row-by-row processing of a multi-row query result set.

Implicit Cursor

Automatically created by Oracle for single-row SELECT INTO, INSERT, UPDATE, DELETE. Accessible via SQL% attributes: SQL%FOUND, SQL%NOTFOUND, SQL%ROWCOUNT.

Explicit Cursor

Defined by the programmer for multi-row queries. Must be explicitly declared, opened (OPEN), fetched row by row (FETCH), and closed (CLOSE).

DECLARE CURSOR emp_cursor IS SELECT name, salary FROM Employee; v_name Employee.name%TYPE; v_sal Employee.salary%TYPE; BEGIN OPEN emp_cursor; LOOP FETCH emp_cursor INTO v_name, v_sal; EXIT WHEN emp_cursor%NOTFOUND; DBMS_OUTPUT.PUT_LINE(v_name || ': ' || v_sal); END LOOP; CLOSE emp_cursor; END;

👆 Imagine a query returns 100 rows. A cursor is like a finger pointing to one row at a time. You OPEN the query result, then FETCH one row, process it, FETCH next row, process it... until done, then CLOSE. It's how PL/SQL handles multi-row results one row at a time, like reading a book page by page instead of the whole book at once.

Views

Textbook Definition

A view is a virtual table defined by a stored SQL query. It does not store data physically (usually); instead, it derives its data from one or more base tables whenever it is accessed. A view presents a customized representation of the data.

-- Creating a view CREATE VIEW HighSalaryEmployees AS SELECT EmpID, Name, Salary, Department FROM Employee WHERE Salary > 50000; -- Using the view like a table SELECT * FROM HighSalaryEmployees WHERE Department = 'CS';

Tables vs Views — Comparison

AspectTableView
StoragePhysically stores data on diskVirtual — stores only the query definition
DataContains actual dataDerives data from base tables at query time
UpdateAlways updatableOnly updatable if it meets certain conditions (no GROUP BY, no DISTINCT, no aggregates)
PerformanceFaster (data already there)Slightly slower (query re-executed each time)

⚠️ Materialized Views are an exception — they physically store the query results for faster access. They need to be refreshed periodically when base data changes.

Packages, Procedures, and Triggers
PL/SQL Advanced▼
Stored Procedure

A stored procedure is a named PL/SQL block stored in the database that performs specific tasks. It can accept parameters, execute SQL and procedural code, and optionally return values. Unlike functions, procedures do not return a value directly.

Parts of a Procedure

CREATE OR REPLACE PROCEDURE procedure_name ( param1 IN datatype, -- Header: name + parameter list param2 OUT datatype, param3 IN OUT datatype ) AS -- Declarative section (local variables) local_var datatype; BEGIN -- Executable section (main logic) SQL statements and PL/SQL code; EXCEPTION -- Exception handling section WHEN OTHERS THEN DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM); END procedure_name;

Parameter Modes

ModeDirectionCan read?Can write?Use case
INCaller → ProcedureYesNo (read-only)Pass input values to procedure
OUTProcedure → CallerNoYesReturn results to caller
IN OUTBoth directionsYesYesPass value, modify it, and return it

Advantages of Stored Procedures

  • Performance: Pre-compiled and cached — faster execution than ad-hoc SQL
  • Security: Users can execute procedure without needing direct table access
  • Reusability: Write once, call from multiple applications
  • Maintainability: Business logic in one place; easier to update
  • Reduced network traffic: One call executes multiple SQL statements

Triggers

Textbook Definition

A trigger is a stored PL/SQL block that is automatically executed (fired) by the database in response to a specified event on a particular table or view. Triggers are implicit — they run without being explicitly called.

CREATE OR REPLACE TRIGGER trigger_name BEFORE | AFTER | INSTEAD OF -- Timing INSERT | UPDATE | DELETE -- DML Event ON table_name FOR EACH ROW | STATEMENT -- Level DECLARE -- declarations BEGIN -- trigger body using :NEW and :OLD pseudo-records IF INSERTING THEN ... ELSIF UPDATING THEN ... END IF; END;

Types of Triggers

ClassificationTypesDescription
By TimingBEFORE, AFTER, INSTEAD OFBEFORE: fires before the DML. AFTER: fires after. INSTEAD OF: fires in place of DML on views.
By EventINSERT, UPDATE, DELETEWhich DML operation triggers it
By LevelROW-level, STATEMENT-levelRow: fires once per affected row. Statement: fires once per statement regardless of rows affected.
By DDLDDL TriggersFire on CREATE, ALTER, DROP operations
System EventsDatabase triggersFire on STARTUP, SHUTDOWN, LOGON, LOGOFF events

In row-level triggers: :NEW.column refers to new values (INSERT/UPDATE), :OLD.column refers to old values (UPDATE/DELETE).

⚡ A trigger is like a security camera alarm. You don't manually trigger the alarm — it fires automatically when something happens (motion detected). Similarly, a DB trigger fires automatically when someone inserts/updates/deletes data. Example: AFTER INSERT on Orders → automatically update inventory count. Or: BEFORE DELETE on Employee → log who was deleted to an audit table. You set it up once and it runs forever automatically.

Packages

Textbook Definition

A package is a schema object that groups logically related PL/SQL types, variables, cursors, constants, subprograms (procedures and functions), and exceptions into a single named unit. A package has two parts: the specification (spec) — the public interface, and the body — the implementation.

-- Package Specification (Public interface) CREATE OR REPLACE PACKAGE emp_pkg AS PROCEDURE hire_employee(name VARCHAR2, dept NUMBER); FUNCTION get_salary(empid NUMBER) RETURN NUMBER; END emp_pkg; -- Package Body (Implementation) CREATE OR REPLACE PACKAGE BODY emp_pkg AS PROCEDURE hire_employee(name VARCHAR2, dept NUMBER) IS BEGIN INSERT INTO Employee ... END; FUNCTION get_salary(empid NUMBER) RETURN NUMBER IS v_sal NUMBER; BEGIN SELECT salary INTO v_sal FROM Employee ... RETURN v_sal; END; END emp_pkg;

Bodiless Package: A package specification with only declarations (constants, types, exceptions) and no body needed — used to define global constants/types shared across the application.

Advantages of Packages

  • Modularity: Encapsulate related procedures/functions together
  • Encapsulation: Hide implementation details; only spec is public
  • Performance: Entire package loaded into memory on first call
  • Overloading: Multiple procedures with same name but different parameters
  • Global variables: Package-level variables persist for the session

Aggregate Functions, NULL Values, and Nested Subqueries

  • Aggregate Functions: COUNT(*), SUM(col), AVG(col), MIN(col), MAX(col) — operate on sets of rows. Used with GROUP BY and HAVING.
  • NULL Values: NULL means unknown/missing, not zero or empty string. NULL ≠ NULL. Use IS NULL / IS NOT NULL. Aggregate functions ignore NULLs (except COUNT(*)). NVL(col, 0) substitutes a value for NULL.
  • Nested Subqueries: A SELECT inside another SQL statement. Can be in WHERE (correlated/non-correlated), FROM (inline view), or SELECT (scalar subquery). Example: SELECT name FROM Employee WHERE salary > (SELECT AVG(salary) FROM Employee);
Transaction Management and Concurrency Control
Critical Topic▼
Textbook Definition

A transaction is a logical unit of work that consists of one or more database operations (read/write) that must be executed as an indivisible unit. Either all operations succeed (COMMIT) or all are undone (ROLLBACK).

ACID Properties

PropertyDefinitionExample
AtomicityAll operations in a transaction complete successfully, or none do. All-or-nothing principle.Bank transfer: debit + credit must both happen, or neither happens.
ConsistencyA transaction brings the database from one valid state to another. All integrity constraints must hold before and after.Account balances sum must remain constant after any transfer.
IsolationConcurrent transactions execute as if they were serial. Intermediate states of a transaction are invisible to others.Two people booking the last seat: one sees the seat booked only after the other's transaction commits.
DurabilityOnce a transaction commits, its effects are permanently recorded even if the system crashes immediately after.After you get "Payment successful," the record persists even through a server crash.

🏦 Bank transfer: ₹10,000 from Account A to Account B. Step 1: Debit A by ₹10,000. Step 2: Credit B by ₹10,000. If the system crashes between steps 1 and 2: Atomicity says UNDO step 1 (rollback). Consistency says total money in bank must stay same. Isolation says another person checking balance during transfer sees either the old or new state, never a half-done state. Durability says after COMMIT, even a power cut won't lose the transfer record (it's in the log).

Serializability and Recoverability

  • Schedule: A sequence of read/write operations from multiple concurrent transactions
  • Serial Schedule: Transactions execute one after another with no interleaving — always correct but slow
  • Serializable Schedule: A concurrent schedule whose effect is equivalent to some serial schedule — correct AND allows concurrency
  • Conflict Serializability: Two operations conflict if they access the same data item, at least one is a write, and they belong to different transactions. A schedule is conflict-serializable if its precedence graph has no cycles.
  • Recoverable Schedule: A transaction T2 commits only after T1 commits if T2 has read data written by T1. Prevents dirty read problems.
  • Cascadeless Schedule: Transactions read only committed data — avoids cascading rollbacks

Concurrency Problems

ProblemDescription
Lost UpdateT1 and T2 both read a value, both update it — T1's update is overwritten by T2 and lost
Dirty ReadT2 reads data written by T1, but T1 later rolls back — T2 used invalid (dirty) data
Unrepeatable ReadT1 reads a value, T2 updates and commits it, T1 reads again and gets a different value
Phantom ReadT1 reads a set of rows, T2 inserts new rows matching T1's query, T1 re-reads and sees new (phantom) rows

Locking Techniques

Shared Lock (S-Lock / Read Lock)

Multiple transactions can hold a shared lock on the same item simultaneously. Allows concurrent reads. If any transaction holds S-lock, no other can get X-lock.

Exclusive Lock (X-Lock / Write Lock)

Only one transaction can hold an exclusive lock. No other transaction can hold any lock (S or X) on that item. Required for writes.

Two-Phase Locking (2PL)

A protocol ensuring serializability. Phase 1 (Growing): Locks are acquired, none released. Phase 2 (Shrinking): Locks are released, none acquired. Guarantees conflict-serializable schedules but can cause deadlocks. Strict 2PL: all locks held until COMMIT — prevents dirty reads.

Deadlock

T1 holds lock on A and waits for B. T2 holds lock on B and waits for A. Neither can proceed — deadlock. Detection via wait-for graph (cycle = deadlock). Prevention: lock ordering, timeout, wound-wait, wait-die schemes.

Timestamp-Based Concurrency Control

Each transaction is assigned a unique timestamp when it starts. Operations are ordered by timestamps instead of locks. Rules: if T tries to read/write a data item already updated by a newer transaction, T is rolled back and restarted with a new timestamp. Avoids deadlocks but may cause many rollbacks.

Optimistic Concurrency Control

Assumes conflicts are rare. Transactions execute without locking. At commit time (validation phase), check if any conflicts occurred. If no conflict → commit; if conflict → rollback and retry. Good for read-heavy workloads with low write conflicts.

Granularity of Data Items

  • Coarse granularity: Database → Table (less overhead, low concurrency)
  • Fine granularity: Row → Attribute (high overhead, high concurrency)
  • Intention Locks: IS, IX, SIX — used in hierarchical locking to signal intent before acquiring fine-grained locks
Database Recovery
Reliability▼
Textbook Definition

Database recovery is the process of restoring the database to a correct, consistent state after a failure. It ensures durability and atomicity of transactions by using recovery techniques that undo uncommitted changes and redo committed changes after a failure.

💾 Imagine you're writing a document and the power cuts. You lose unsaved changes. Database recovery prevents this. The DBMS continuously maintains a "diary" (transaction log) of every change made. If the system crashes, it reads the diary: "Did this transaction commit? Yes → redo it. Did this one commit? No → undo it." It's like Ctrl+Z with a perfect history that even survives crashes.

Types of Failures

Failure TypeDescriptionRecovery Method
Transaction FailureLogical error (division by zero, constraint violation) or system error causing rollback. Only affects one transaction.Rollback using transaction log (UNDO)
System Failure (Crash)OS crash, power failure, DBMS bug. Main memory (buffer) is lost; disk data survives. Affects all active transactions.Log-based recovery (UNDO uncommitted, REDO committed)
Media FailureDisk head crash or storage corruption. Disk data is lost or damaged. Most severe failure.Restore from backup + replay log from last backup

Log-Based Recovery

Write-Ahead Log (WAL) Protocol

Before any data item is written to the database, the log record describing the change must be written to stable storage. This ensures the log always has enough information to undo or redo changes.

  • Log Record: [Transaction ID, Data item, Old Value (BFIM), New Value (AFIM), Operation type]
  • UNDO (Rollback): Use old values (BFIM - Before Image) to reverse uncommitted transactions
  • REDO: Use new values (AFIM - After Image) to re-apply committed transactions lost in crash
  • COMMIT record: Written when transaction commits — presence in log = committed; absence = must undo

Checkpoint-Based Recovery

The DBMS periodically writes a checkpoint to the log: forces all dirty buffer pages to disk and records the set of active transactions. On recovery, only scan the log from the last checkpoint — not from the beginning. This greatly reduces recovery time.

Recovery after crash using checkpoints: 1. Find last checkpoint record → list of active transactions at that point 2. Scan forward from checkpoint to crash point 3. Divide transactions into: - Redo list: had COMMIT record → redo their changes - Undo list: no COMMIT record → undo their changes 4. Apply REDOs forward, apply UNDOs backward

Shadow Paging

An alternative to log-based recovery. Maintains two page tables: current and shadow. The shadow page table always points to the last committed state. During a transaction, new pages are written to new disk blocks; the shadow is untouched. On commit, the current table replaces the shadow atomically. On failure, discard current table and use shadow — instant rollback. Disadvantage: poor performance due to random I/O patterns.

Recovery Management Process

  1. Detection: Detect the failure (system crash, disk error, etc.)
  2. Assessment: Determine the type and extent of failure
  3. Analysis: Scan the log to identify committed and uncommitted transactions
  4. REDO: Re-apply all changes of committed transactions (forward recovery)
  5. UNDO: Reverse all changes of uncommitted transactions (backward recovery)
  6. Verification: Ensure database is in a consistent state
  7. Resumption: Resume normal database operations
